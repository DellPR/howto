01 · The Starting Point

It started with one sentence.

A basic Word document. A travel guide for journalists. One ask: make it better.

The original file

📄 GUIA DE VIAGEM — AUSTIN PRESS TOUR
✈️  Voos (flight screenshots)
🏨  Hospedagem — Hyatt Place
📅  Agenda (plain text list)
📍  Tour descriptions
👤  Spokesperson bios

→ A Word doc. Static. Not mobile-friendly.

The actual first prompt

💬 User → Claude
I'm working on a travel guide for journalists we're taking to Austin for a press tour. Could you take the attached guide and make it more beautiful, efficient and modern — and also add things like a clickable index with links? We want this to be the journalists' bible for the trip.

+ attached Guia.docx

What actually came back was an improved version of the same Word document — cleaner, with a proper index and better structure. A good upgrade, but still a static doc.

That's when the real brief happened. Thinking about who would actually use this — a journalist on their phone, in transit, trying to find a schedule or a venue — it was clear a document wasn't enough. The next prompt described exactly that context: who the user was, what they needed to find fast, and that it had to work on mobile.

From there, hundreds of prompts followed — refining layouts, adding sections, tuning the visual identity, personalizing for each journalist. The Word doc was just the starting line.

03 · The Workflow

5 repeatable steps.

Once the first app was built and fine-tuned, all 6 personalized versions were generated in one shot — just by pasting each journalist's flight info and details. Claude handled the rest automatically.

01
Start with the output in mind
Before opening Claude, define: Who will use this? On what device? What must they find in 10 seconds?
→ "A journalist on their phone during the trip. Needs schedule + spokesperson info fast."
02
Write the brief as a creative director
Your first prompt sets everything. Include audience, tone, brand, structure, and sample data.
→ "Create a mobile-first HTML app for Wallace Mote from TudoCelular. Dell blue #0076CE. IBM Plex font. Apr 15 schedule."
03
Iterate visually
Open the file. Screenshot what's wrong. Describe it plainly. One round takes under a minute.
→ "Move the spokesperson section above the schedule. Increase the header font. Remove the footer line."
04
Drop in the real content
Once the structure is right, paste in actual bios, schedules, specs. AI reformats to match the design.
→ "Here is Wallace's confirmed bio and his Apr 15-17 schedule. Add these to the existing template."
05
Publish and keep iterating
Upload HTML to GitHub Pages. Share the URL. Any update is a new upload. No CMS, no developer.
→ "The Wednesday schedule changed. Update the 14:30 session and regenerate the file."

05 · The Repository Idea

One more problem to solve.

The apps covered the trip logistics. But the journalists would also need background — on Dell AI, on TACC, on the products. That content needed a home.

THE PROBLEM

Press tours generate a lot of supporting material — product specs, press releases, customer stories, context on the venues. Sharing this as PDFs or email attachments meant journalists would lose it, ignore it, or not have it when they needed it.

THE INSIGHT

The content needed to be accessible, updatable, and adapt to whatever screen the journalist was on — a phone between sessions, a laptop at the hotel, a tablet on the plane. A hosted web page was the obvious answer.

THE BRIEF THAT STARTED IT
💬 User → Claude
"I want to create a repository page where I can share articles and content about the trip topics — Dell AI, TACC, the products we'll be showing. Journalists will access it from different devices so it needs to be fully responsive. I want to be able to update it and have them always see the latest version. It should match the visual style of the journalist apps."

06 · Installable as a Native App

Add to Home Screen — it looks like a real app.

Each journalist could install their app directly from the browser. It got its own icon on the home screen, launched full-screen, and worked offline.

The apps were built as Progressive Web Apps (PWA). A single prompt added the manifest and install instructions. When journalists followed them, the app appeared on their phone's home screen — indistinguishable from a native app downloaded from the App Store.

💬 The prompt that made it happen
"Add a manifest.json and the necessary meta tags so journalists can install this as a web app on their iPhone. When they tap Add to Home Screen it should show the Dell Technologies icon and open full-screen with no browser chrome."

08 · Writing Prompts That Work

Specificity is the skill. Not programming.

✗ Vague — generic result
"Make me an app for the press trip."
✓ Specific — exactly what you need
"Create a mobile-first HTML file for Rodrigo Riquelme, journalist at El Economista in Mexico. Show his April 15 schedule with times, the 3 spokespeople he will meet with their bios and LinkedIn links, and the restaurant details for dinner. Use Dell blue #0076CE, white cards on light grey, IBM Plex font."
🎯
Name the audience precisely
Don't say "users". Say "a journalist on their phone during the trip, in a cab."
📐
Describe the layout visually
"Two columns. Blue header. White cards on light grey. Schedule on the left."
🎨
Use actual brand values
"Dell Blue #0076CE. IBM Plex font. Same style as the previous app."
🔄
Iterate with precision
"Move section 3 up. Follow the same Dell brand guidelines from the previous app. Remove the footer."
📋
Paste the real data in
Drop in the actual bio, schedule, or spec list. Let AI format it to match the design.
🌐
Always name the output format
"Generate this as a single self-contained HTML file I can upload directly."
Try it yourself.

Open a conversation. Describe what you want to build. The first good output comes in 3–5 prompts.
